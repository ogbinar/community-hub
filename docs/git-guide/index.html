What Is Version Control?

Version control is a system that records changes to your files over time. Think of it like "Save As" but smarter — it lets you go back to any previous version, see who changed what, and work on multiple versions simultaneously.

Git is the version control system. GitHub is a platform that hosts your Git repositories online — like Google Drive for code.

Why This Matters for Data Professionals

  • 📋 Employers expect data professionals to use Git (58.6% of ₱100k+ earners do)
  • 📋 It's the #1 tool for collaboration — sharing code with teammates
  • 📋 A GitHub portfolio is more impressive than a resume alone

Getting Started: Step-by-Step

  1. Go to github.com and create a free account. Use your real name.
  2. Click "New Repository" (green "New" button in top right).
  3. Name it something descriptive: e.g., my-first-data-project.
  4. Check "Add a README file" and click "Create repository".

The Full Commit Workflow

# 1. Check what changed
git status

# 2. Stage the files you want to commit
git add analysis.py

# 3. Commit with a descriptive message
git commit -m "feat: add data analysis script"

# 4. Push to GitHub (first time only requires setup)
git push origin main

Branching Strategies

Branches let you work on new features without breaking your main project. Think of it like making a copy of your workbook to experiment — if something goes wrong, your original is still safe.

Feature Branch Workflow (Recommended for Beginners)

# Create a new branch and switch to it
git checkout -b feature/add-customer-segmentation

# Make your changes, then commit
git add .
git commit -m "feat: add customer segmentation query"

# Push the branch to GitHub
git push origin feature/add-customer-segmentation

Visual:

main:    O---O---O---O---O
              \
feature:       O---O---O

Commit Conventions

Use conventional commit prefixes to make your history readable:

Prefix When to Use Example
feat: New feature or module "feat: add customer segmentation query"
fix: Bug fix or correction "fix: correct date format in ETL pipeline"
docs: Documentation changes "docs: update README with setup instructions"
refactor: Code restructuring "refactor: simplify data cleaning function"

Pull Requests (PR) Workflow

  1. Push your feature branch to GitHub (see Step 3 above).
  2. Go to your repo on github.com — you'll see a banner: "Branch X had recent pushes" with a green "Compare & pull request" button.
  3. Click the button — GitHub will show a diff (what changed).
  4. Add a title and description explaining what you changed and why.
  5. Click "Create pull request" — this asks your team to review your changes before merging them into the main codebase.

PR Template (Copy-Paste This)

## What I Changed
Brief description of the changes made.

## Why I Changed It
The reason for this change (e.g., fixing a bug, adding a feature).

## How to Test
Steps to verify the changes work correctly.

Portfolio Showcase Guide

What Makes a Great GitHub Portfolio

  • Clear README with project description, tools used, what you learned, and screenshots of results.
  • At least 2–3 complete projects (not just tutorial walkthroughs).
  • Clean file structure with organized folders (src/, data/, notebooks/).
  • Code comments explaining key logic in your scripts.
  • Link to GitHub in your LinkedIn "Featured" section.

Recommended Project Structure

my-data-project/
├── README.md          # Project description, tools, results
├── src/               # Source code
│   ├── data_clean.py  # Data cleaning scripts
│   └── analysis.py    # Analysis scripts
├── notebooks/         # Jupyter notebooks
│   └── exploratory.ipynb
├── data/              # Raw and processed data (add to .gitignore)
│   ├── raw.csv
│   └── cleaned.csv
├── output/            # Dashboard exports, reports
│   └── dashboard.pbix
└── requirements.txt   # Python dependencies
