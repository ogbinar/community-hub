Git & GitHub Adoption Guide

A step-by-step guide for data professionals to adopt version control. Covers: GitHub basics, branching strategies, commit conventions, and PR workflows.

🔎 Why Git Matters for Data Work

Only 48.8% of the community uses version control. If you want to stand out in job applications, Git is non-negotiable.

What Git Does

  • Track changes — Every edit is saved with a timestamp and description
  • Undo mistakes — Go back to any previous version of your code
  • Collaborate — Multiple people can work on the same project without overwriting each other
  • Showcase work — Your GitHub profile becomes a public portfolio of projects

📄 GitHub Basics

Step 1: Create a Repository

A repository (repo) is like a folder for your project. Every GitHub project has its own repo.

# From the command line (requires Git installed)
git init my-first-project
cd my-first-project
git add .
git commit -m "Initial project setup"
git branch --initial-branch=main
git remote add origin https://github.com/yourusername/my-first-project.git
git push -u origin main

Or use the GitHub web interface: Click "New Repository" → Name it → Click "Create repository".

Step 2: Write a README.md

A README file is the first thing recruiters see. It explains what your project does, how to use it, and what tools you used.

# Project Title

## Overview
Brief description of what this project does and why it matters.

## Tools Used
- Python pandas for data manipulation
- SQL for data extraction
- Power BI for visualization

## How to Run
1. Clone the repository
2. Install dependencies: pip install -r requirements.txt
3. Run: python main.py

Save this as README.md in your project folder.

📍 Branching Strategies

Branches let you work on new features without breaking the main project. Think of a branch as a parallel timeline for your code.

Simple Branching Workflow

# Create a new branch for your feature
git checkout -b add-new-analysis

# Make changes and commit them
git add .
git commit -m "Add new analysis script"

# Switch back to main
git checkout main

# Merge your changes into main
git merge add-new-analysis

Best practice: Never push directly to the main branch. Always create a feature branch, make your changes, then merge.

📋 Commit Conventions

Good commit messages help you and others understand what changed. Follow this pattern:

# Good commit messages
git commit -m "Add: ETL pipeline for sales data"
git commit -m "Fix: Handle missing values in customer table"
git commit -m "Update: Improve dashboard performance"

# Bad commit messages (don't do this)
git commit -m "update"
git commit -m "fix stuff"
git commit -m "changes"

Convention: Start with a verb (Add, Fix, Update, Remove) followed by a brief description of what changed.

📅 Pull Request (PR) Workflows

Pull Requests let you propose changes to a project and invite others to review them before merging.

PR Checklist

  1. 1
    Create a feature branch for your changes
  2. 2
    Push your branch to GitHub
  3. 3
    Open a Pull Request on GitHub with a clear description
  4. 4
    Ask a mentor or peer to review your code
  5. 5
    Merge after approval
# Push your branch to GitHub
git push -u origin add-new-analysis

# Then go to github.com and click "Compare & pull request"
